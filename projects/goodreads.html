The Goodreads Search Bar

Redesigning a web component for accessibility and usability

Role

Interaction Designer

Timeline

February 2025

Skills

Figma • Analysis

Overview

Webpages and digital interfaces are becoming more and more ubiquitous. However, too often this new way of living is not designed with accessibility in mind—websites frequently forego accessibile designs in exchange for ones that are more visually appealing, or don't put in the effort to include accessible features like voiceover or alt text.

In this case study, we will focus on the Goodreads search bar, which is a common feature on the website. We will explore how the search bar can be redesigned to be more accessible to users with vision, hearing, and motor impairments.

Problem

How can we design a search bar that works for everyone?

Research

First, I chose three applications—Goodreads, VSCode, and Notion—and analyzed their existing search bars. I focused on the question:

What are their current features, and how do they impact accessibility?

Goodreads

Pros

  1. Accessibility: best voiceover implementation-clearly communicates functionality, input types, and content.
  2. Clearly indicates input options (e.g., Author, Title, ISBN) for users.

Cons

  1. Learnability: lack of visual feedback (e.g. hover cues or shading) make it hard to learn or distinguish states.
  2. Efficiency: No keyboard shortcuts and poor keyboard navigation reduce efficiency.
  3. "Enter" key and arrow keys do not activate important actions, breaking expected functionality.

Default Goodreads search bar - no change on hover

On click, search bar is outlined with shadow

VSCode

Pros

  1. Discoverability: Strong keyboard support with clearly shortcuts (e.g., ⌘P, ⌘⇧P).
  2. Learnability: Helpful visual cues on mouse hover (shading, cursor change).
  3. Interface is intuitive to navigate via keyboard once inside the main UI.

Cons

  1. Accessibility: Poor voiceover support–labels are unclear or incorrect.
  2. Tabbing gets stuck when navigating to embedded components like the terminal.
  3. No clear feedback or labeling on click/release for accessibility tools.

Notion

Pros

  1. Learnability & Discoverability: Many descriptive hover dialogs and shortcut hints.
  2. Strong visual feedback—shading and pop-ups clearly communicate state changes.
  3. Accessibility: Voiceover accurately identifies search input and basic functionality.

Cons

  1. Accessibility: Unintuitive and inefficient keyboard navigation through the sidebar.
  2. Accessibility: Shortcut text is not read aloud by voiceover, reducing clarity and efficiency.

After analyzing the three applications, I found Goodreads's search bar to have the most issues. Additionally, considering the context of Goodreads' user base of readers, I wanted to explore redesigning the search bar to be more accessible to that user group.

Final Redesign

Finally, I created a final redesign of the search bar, incorporating all the changes I made to the state models. I also added a new microphone icon to allow for voice input.

Tradeoffs:

I prioritized accessibility and visual clarity for my new design. However, this does come at the cost of simplicity—the new design has more states and interactions, which may be overwhelming for some users or detract from memorability.

Conclusion

Findings

While some accessibility considerations were present, most components performed only moderately well for each example, with significant gaps—particularly in screen reader support and input flexibility. Mouse and touch inputs were clearly prioritized across platforms, leaving users who rely on keyboard or voice input underserved. Surprisingly, none of the search bars supported non-text (e.g. voice, image) input, and screen reader functionality in tools like VSCode was especially lacking.

To address these accessibility mismatches in my design, I implemented features like a keyboard shortcut dialog and voice input support in order to accommodate a broader range of users.

Next Steps

This case study has definitely been an eye-opening experinece to the lack of accessibility in the digital world. I now approach my web interactions with far more awareness of these issues than I did before—and there are many issues to notice!

Going forward, I will make an effort to focus on supporting multiple input modalities equally in my designs—mouse, touch, keyboard, voice, and screen readers—to create inclusive and efficient navigation for all users, including those with disabilities or age-related challenges.
